# Beta Launch Guide — Mastery Engine Closed Beta (20 Users)

> **Audience:** Platform administrator (the person who creates invites, manages users, and triages feedback).
> **Prerequisite:** `production-launch.md` is complete — platform is live at `https://app.masteryengine.com`.
> **Last updated:** 2026-07-03

## 0. What This Guide Covers

This is the **operational playbook** for running the Closed Beta. It covers:

1. Admin onboarding (your first login, security setup)
2. Invite creation and management
3. First-user onboarding experience
4. Feature flag management
5. Feedback triage workflow
6. Analytics review
7. User limit management

It does **not** cover infrastructure operations — see `operations-checklist.md` for that.

## 1. Admin Onboarding

### 1.1 First login

You should already have an admin account created during `production-launch.md` §7.1. If not, ask the SRE to bootstrap one.

1. Open `https://app.masteryengine.com/login` in a modern browser (Chrome, Firefox, Safari, or Edge — last 2 major versions).
2. Log in with `admin@masteryengine.com` and the password from the secrets store.
3. You will be prompted to set up MFA (TOTP). Scan the QR code with an authenticator app (1Password, Authy, Google Authenticator, Microsoft Authenticator).
4. Save the **10 recovery codes** displayed. Store them in a password manager or print and store in a physical safe. Each code is single-use.
5. You're now on the admin dashboard.

### 1.2 Verify admin access

| Page | URL | What to verify |
|---|---|---|
| Operations Dashboard | `/admin` | Charts load, no error toasts |
| Users | `/admin/users` | You see yourself in the list with role `administrator` |
| Organizations | `/admin/organizations` | Empty list (no orgs yet for beta) |
| RBAC | `/admin/rbac` | You see the role hierarchy |
| Feature Flags | `/admin/feature-flags` | 6 beta flags visible, all matching `.env.production` |
| Workers | `/admin/workers` | At least 1 worker `running`, `last_seen` < 30s ago |
| Scheduler | `/admin/scheduler` | List of scheduled jobs with next_run_at |
| Outbox | `/admin/outbox` | Pending count = 0 (or trending to 0) |
| Dead Letters | `/admin/dead-letters` | Empty |
| Email | `/admin/email` | Recent delivery log entries |
| Audit | `/admin/audit` | Your login event is the most recent entry |
| Security | `/admin/security` | No active security incidents |
| Beta Invites | (via API only — no UI yet) | See §2 below |

### 1.3 Security hardening for your admin account

- [ ] **Use a unique 16+ character password** generated by a password manager.
- [ ] **MFA is mandatory** — do not skip the recovery codes step.
- [ ] **Do not share your admin credentials** with anyone. If a colleague needs admin access, create a separate account for them.
- [ ] **Review the audit log weekly** for suspicious activity on your account (`/admin/audit?user_id=<your-id>`).
- [ ] **Change your password immediately** if you suspect any compromise.

### 1.4 Add a second admin (recommended)

For beta resilience, add a second admin so you have coverage during vacations or incidents:

```bash
# 1. Create an invite for the second admin (beta is enabled, so invite required)
curl -s -X POST https://app.masteryengine.com/api/v1/admin/beta/invites \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin2@masteryengine.com","notes":"Backup admin"}' \
  | jq -c '{email: .email, token: .invite_token}'

# 2. They register using the invite link
# 3. Promote them to administrator
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
  "UPDATE identity.users SET role='administrator' WHERE email='admin2@masteryengine.com';"

# 4. They log in, set up MFA, save recovery codes
```

## 2. Invite Management

### 2.1 The invite lifecycle

```
[Created] → [Valid (7 days)] → [Used] (registration complete)
                              ↘ [Expired] (TTL elapsed, never used)
                              ↘ [Deleted] (admin removed unused invite)
```

Once used, an invite cannot be reused or deleted. Once expired, it must be resent (generates a new token, extends TTL by 7 days).

### 2.2 Creating a single invite

**Via API** (no UI yet):

```bash
curl -s -X POST https://app.masteryengine.com/api/v1/admin/beta/invites \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{
    "email": "newuser@example.com",
    "notes": "Friend of referrer X, Python intermediate"
  }' | jq .
```

Response:
```json
{
  "id": "uuid",
  "email": "newuser@example.com",
  "invite_token": "32-char-url-safe-token",
  "expires_at": "2026-07-10T12:00:00Z",
  "used_at": null,
  "created_by": "your-admin-uuid",
  "notes": "Friend of referrer X, Python intermediate",
  "is_used": false,
  "is_expired": false,
  "is_valid": true
}
```

**The invite URL to send to the user:**

```
https://app.masteryengine.com/register?invite_token=<invite_token>&email=<email>
```

> ⚠️ **Email dispatch is not yet automated.** You must manually send the invite URL to the user via email, Slack, or your preferred channel. See `production-launch.md` §7.2 for how to wire automated email dispatch.

### 2.3 Creating multiple invites (batch)

```bash
# Save this as /tmp/invite_emails.txt (one email per line)
cat > /tmp/invite_emails.txt <<EOF
user1@example.com
user2@example.com
user3@example.com
EOF

# Loop and create
while read email; do
  result=$(curl -s -X POST https://app.masteryengine.com/api/v1/admin/beta/invites \
    -H "Authorization: Bearer $ADMIN_TOKEN" \
    -H 'Content-Type: application/json' \
    -d "{\"email\":\"$email\",\"notes\":\"Closed Beta batch 2\"}")
  token=$(echo "$result" | jq -r '.invite_token')
  echo "$email,$token,https://app.masteryengine.com/register?invite_token=$token&email=$email"
done < /tmp/invite_emails.txt | tee /tmp/invite_links.csv
```

The CSV at `/tmp/invite_links.csv` contains one row per invite with the email, token, and full URL. Use this to send personalized emails.

### 2.4 Listing invites

```bash
# All invites (including used)
curl -sf "https://app.masteryengine.com/api/v1/admin/beta/invites?include_used=true" \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq -r '.[] | "\(.email)\t\(.is_used)\t\(.is_expired)\t\(.expires_at)"'

# Only unused invites (default)
curl -sf "https://app.masteryengine.com/api/v1/admin/beta/invites" \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq -r '.[] | "\(.email)\t\(.expires_at)"'
```

### 2.5 Resending an expired invite

If a user didn't use their invite within 7 days, the token has expired. Resend:

```bash
# 1. Find the invite ID
INVITE_ID=$(curl -sf "https://app.masteryengine.com/api/v1/admin/beta/invites?include_used=true" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  | jq -r '.[] | select(.email=="user@example.com") | .id')

# 2. Resend (generates a new token, extends TTL by 7 days)
curl -s -X POST https://app.masteryengine.com/api/v1/admin/beta/invites/resend \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"invite_id\":\"$INVITE_ID\"}" | jq -c '{email: .email, new_token: .invite_token, new_expiry: .expires_at}'
```

The old token is invalidated; the new token is valid for 7 more days.

### 2.6 Deleting an unused invite

If an invite was created in error or the user changed their mind:

```bash
# Only unused invites can be deleted
curl -s -X DELETE "https://app.masteryengine.com/api/v1/admin/beta/invites/$INVITE_ID" \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq .
# Expected: {"message":"Invite deleted","code":"OK"}
```

If the invite has already been used, the API returns 404 — used invites are immutable.

### 2.7 Tracking invite usage

Daily check:

```bash
# Beta status
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq .
# Output: closed_beta_enabled, max_beta_users, current_user_count, feature_flags

# Invites summary
curl -sf "https://app.masteryengine.com/api/v1/admin/beta/invites?include_used=true" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  | jq '{
      total: length,
      used: [.[] | select(.is_used)] | length,
      unused: [.[] | select(.is_used == false)] | length,
      expired: [.[] | select(.is_expired)] | length
    }'
```

## 3. First-User Onboarding Experience

When a beta user accepts their invite, here's what they experience. Knowing this flow helps you support them.

### 3.1 The registration flow

1. User clicks the invite URL: `https://app.masteryengine.com/register?invite_token=<token>&email=<email>`
2. The register page pre-fills the email field.
3. User enters a password (must meet strength requirements — see password strength meter).
4. User clicks "Create account".
5. The backend validates:
   - Beta is enabled (`CLOSED_BETA_ENABLED=true`)
   - User count < 20 (`MAX_BETA_USERS`)
   - Token exists in `beta_invites`
   - Token is not yet used (`used_at IS NULL`)
   - Token is not expired (`expires_at > now()`)
   - Email matches the invite email (case-insensitive)
6. On success: account created, invite marked as used, welcome email queued, user logged in.
7. On failure: HTTP 403 with a descriptive error message (`BETA_REGISTRATION_DENIED`, `BETA_FULL`, `INVITE_EXPIRED`, etc.)

### 3.2 The welcome wizard

After first login, the user lands on `/welcome` — a 4-step wizard:

| Step | What the user does | Notes |
|---|---|---|
| 1. Profile | Enters display name and timezone | Timezone is used for daily-streak calculations |
| 2. Learning Goal | Picks daily question goal (5/10/15/20) | Affects recommendation pacing |
| 3. Subject | Picks a subject (Python Interview Prep / Data Structures / Algorithms) | Determines initial content pack |
| 4. Tutorial | Brief walkthrough of the UI | "Get Started" → `/dashboard` |

> ⚠️ **Known issue**: the wizard does not currently persist data to the backend. The user's display name, timezone, goal, and subject selections are discarded on completion. Track this as a P2 fix. Workaround: the user can update these later in `/settings`.

### 3.3 The beta banner

Every authenticated learner page shows a thin banner at the top:

> 🧪 Closed Beta · v1.0.0 · production

This sets the expectation that the platform is in beta and may have rough edges.

### 3.4 The feedback button

Every authenticated learner page shows a floating button at bottom-right:

> 💬 Feedback

Clicking it opens a dialog with:
- 5-star rating (1–5)
- Category dropdown: Bug, Feature Request, UI/UX, Content, Performance, Other
- Comment field (1–5000 chars)
- Auto-captured context: browser, platform, current route, correlation ID, user agent

On submit, the backend stores the feedback in `identity.beta_feedback` with status `open`.

### 3.5 The user's first study session

Recommended onboarding path for a new beta user:

1. Complete the welcome wizard (5 min)
2. Open `/dashboard` — should see empty state with "Start your first study session" CTA
3. Click the CTA → `/study/start`
4. Pick a subject → start a session
5. Answer 5–10 questions of varying types
6. End the session → see summary
7. Open `/mastery` — should see updated mastery scores
8. Open `/recommendations` — should see next-step suggestions
9. Click the floating feedback button → submit their first impression

This entire flow should take 15–20 minutes. If a user gets stuck anywhere, that's a UX signal worth investigating.

## 4. Feature Flag Management

### 4.1 The 6 beta flags

| Flag | What it controls | Default for beta |
|---|---|---|
| `BETA_FLAG_LEARNING_ENABLED` | Study sessions, questions, mastery tracking | `true` |
| `BETA_FLAG_CONTENT_AUTHORING_ENABLED` | Content authoring portal (`/content/*`) | `true` |
| `BETA_FLAG_AI_ENABLED` | AI explanations, AI coach, AI feedback | `false` |
| `BETA_FLAG_NOTIFICATIONS_ENABLED` | Email + in-app notifications | `true` |
| `BETA_FLAG_ANALYTICS_ENABLED` | Event tracking, analytics dashboard | `true` |
| `BETA_FLAG_ADMIN_CONSOLE_ENABLED` | Admin portal (`/admin/*`) | `true` |

### 4.2 Reading current flags

```bash
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq '.feature_flags'
```

The frontend polls this endpoint at runtime, so changes take effect for users on their next page load (no cache invalidation needed on the client).

### 4.3 Changing a flag

Flags are environment variables read by the `Settings` class on backend startup. They are **NOT hot-reloadable** — changing them requires restarting both `backend` and `worker`.

```bash
# 1. Edit .env.production on the server
sudo nano /opt/mastery-engine/.env.production
# Change, e.g., BETA_FLAG_AI_ENABLED=true

# 2. Restart both services
cd /opt/mastery-engine
docker compose --env-file .env.production -f docker-compose.prod.yml restart backend worker

# 3. Wait for healthchecks (30s)
docker compose -f docker-compose.prod.yml ps

# 4. Verify the new value
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq '.feature_flags.ai_enabled'
# Expected: true
```

### 4.4 Common flag-change scenarios

#### Scenario A: AI isn't ready for beta yet, leave it off

- Keep `BETA_FLAG_AI_ENABLED=false` for the first 2 weeks.
- After collecting feedback on the core learning experience, evaluate whether AI adds value.
- If yes: provision Ollama or OpenAI, set the env vars, flip the flag, restart.

#### Scenario B: A subsystem is causing incidents, kill it temporarily

If notifications are causing worker backpressure:

1. Set `BETA_FLAG_NOTIFICATIONS_ENABLED=false`
2. Restart backend + worker
3. The frontend will hide the notification bell; the backend will skip notification handlers.
4. Diagnose and fix the underlying issue (see `troubleshooting.md`).
5. Re-enable.

#### Scenario C: Expanding the user base

After beta stabilises, raise `MAX_BETA_USERS`:

```bash
# Edit .env.production: MAX_BETA_USERS=50
docker compose --env-file .env.production -f docker-compose.prod.yml restart backend worker

# Verify
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq '.max_beta_users'
# Expected: 50
```

Note: before expanding, review capacity (DB connections, Redis memory, worker throughput) per `operations-checklist.md`.

## 5. Feedback Triage Workflow

### 5.1 Reading feedback

```bash
# All open feedback
curl -sf "https://app.masteryengine.com/api/v1/beta/feedback?status_filter=open" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  | jq -r '.[] | "\(.created_at)\t\(.rating)\t\(.category)\t\(.comment[:80])"'

# Feedback by category
curl -sf "https://app.masteryengine.com/api/v1/beta/feedback" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  | jq -r '.[] | select(.category=="bug") | "\(.id)\t\(.rating)\t\(.comment[:100])"'

# Feedback from a specific user
curl -sf "https://app.masteryengine.com/api/v1/beta/feedback" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  | jq -r '.[] | select(.user_id=="<uuid>") | "\(.created_at)\t\(.rating)\t\(.comment)"'
```

### 5.2 Triage categories

| Category | Typical response time | Action |
|---|---|---|
| Bug | 24 hours | Reproduce; if confirmed, file in tracker; respond to user with fix ETA |
| Feature Request | 1 week | Acknowledge; add to backlog; respond with prioritisation |
| UI/UX | 3 days | Acknowledge; add to design review; respond with intent |
| Content | 2 days | Review content; fix or flag to content author; respond |
| Performance | 2 days | Reproduce; profile; file perf ticket; respond with investigation status |
| Other | 1 week | Acknowledge; route to appropriate owner |

### 5.3 Updating feedback status

The feedback API currently supports read operations only. To update status, use direct SQL:

```bash
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  UPDATE identity.beta_feedback
  SET status='acknowledged', updated_at=now()
  WHERE id='<feedback-uuid>';
"

docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  UPDATE identity.beta_feedback
  SET status='resolved', resolved_at=now(), resolved_by='<admin-uuid>',
      resolution_notes='Fixed in v1.0.1 — the question type renderer was missing a null check.'
  WHERE id='<feedback-uuid>';
"
```

Status values: `open`, `acknowledged`, `resolved`, `closed`.

### 5.4 Feedback analytics

```bash
curl -sf https://app.masteryengine.com/api/v1/beta/analytics \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq .
```

Response includes:
- Total feedback count
- Average rating
- Rating distribution (1-star to 5-star counts)
- Category breakdown
- Status breakdown
- Recent feedback list

### 5.5 Closing the loop with users

When a bug reported via feedback is fixed:

1. Find the user's email:
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
     "SELECT u.email FROM identity.users u
      JOIN identity.beta_feedback f ON f.user_id=u.id
      WHERE f.id='<feedback-uuid>';"
   ```
2. Email the user: "You reported [issue] on [date]. We've fixed it in v1.0.1. Thanks for the feedback — it directly shaped the platform."
3. Mark the feedback as `resolved` with resolution notes.

This closes the loop and dramatically increases user retention for beta programs.

## 7. User Limit Management

### 7.1 What happens when the beta is full

When `current_user_count == MAX_BETA_USERS`:

- New registration attempts return `403 BETA_FULL`
- The frontend shows: "Closed Beta is currently full. New invites will be available soon."
- Unused invites remain valid — they can still be used if a user unregisters (not currently supported)

### 7.2 Monitoring user count

```bash
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq '{current: .current_user_count, max: .max_beta_users}'
```

Set up an alert when `current >= max - 2` (2 slots remaining):

```bash
# Add to alerts.yml (see post-launch-monitoring.md §3)
- alert: BetaAlmostFull
  expr: 20 - mastery_beta_user_count > 0 and 20 - mastery_beta_user_count < 3
  for: 1m
  annotations:
    summary: "Closed Beta is almost full ({{ $value }} slots remaining)"
```

(Note: this requires a custom Prometheus metric — see `post-launch-monitoring.md` §5.)

### 7.3 Expanding the beta

When you're ready to grow beyond 20 users:

1. Capacity review (see `operations-checklist.md`):
   - [ ] DB connections averaging < 50% of `max_connections`
   - [ ] Redis memory < 70% of `maxmemory`
   - [ ] Worker throughput keeping up with outbox
   - [ ] No SEV-1/SEV-2 incidents in the last 2 weeks
2. Decide new cap (e.g. 50).
3. Update `.env.production`: `MAX_BETA_USERS=50`.
4. Restart backend + worker.
5. Create new invites.
6. Announce to stakeholders.

### 7.4 Removing inactive users (not currently supported)

There is no automated way to remove a registered user. If you need to free up a slot:

```bash
# 1. Identify the inactive user (e.g. last login > 30 days ago)
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  SELECT u.id, u.email, u.last_login_at
  FROM identity.users u
  WHERE u.role != 'administrator'
  ORDER BY u.last_login_at ASC
  LIMIT 5;
"

# 2. Soft-delete by revoking their tokens (recommended)
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  -- Revoke all refresh tokens
  UPDATE identity.refresh_tokens
  SET revoked_at=now(), revoke_reason='Account deactivated (inactive)'
  WHERE user_id='<uuid>' AND revoked_at IS NULL;

  -- Revoke all sessions
  UPDATE identity.sessions
  SET revoked_at=now(), revoke_reason='Account deactivated (inactive)'
  WHERE user_id='<uuid>' AND revoked_at IS NULL;

  -- Mark user inactive (no schema support for soft delete — track in notes)
"

# 3. Notify the user via email that their account was deactivated due to inactivity
# 4. They cannot log in again unless re-invited (their invite token is already used)
# 5. The current_user_count does NOT decrease — this is a known limitation.
```

Track "delete user" as a P1 feature for post-beta.

## 9. Transitioning Out of Beta

When you're ready to open the platform to the public (or to a larger cohort):

### 9.1 Pre-transition checklist

- [ ] Closed Beta ran for at least 6 weeks
- [ ] All P1 bugs resolved
- [ ] User retention > 60% (week-over-week)
- [ ] Average rating > 4.0/5
- [ ] No SEV-1 incidents in the last 4 weeks
- [ ] Capacity validated for 5x current load (load test)
- [ ] Backup/restore drill completed
- [ ] Security audit passed
- [ ] Public documentation (help center, FAQ) written

### 9.2 The transition

```bash
# 1. Disable closed beta
sed -i 's/CLOSED_BETA_ENABLED=true/CLOSED_BETA_ENABLED=false/' /opt/mastery-engine/.env.production
docker compose -f /opt/mastery-engine/docker-compose.prod.yml restart backend worker

# 2. Verify
curl -sf https://app.masteryengine.com/api/v1/beta/status | jq '.closed_beta_enabled'
# Expected: false

# 3. Test public registration (no invite required)
```

## 10. Common Admin Scenarios

### 10.1 "A user can't log in"

1. Check if their account exists:
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
     "SELECT id, email, email_verified, role, last_login_at FROM identity.users WHERE email='user@example.com';"
   ```
2. If email is not verified, resend verification:
   ```bash
   curl -X POST https://app.masteryengine.com/api/v1/auth/resend-verification \
     -H 'Content-Type: application/json' \
     -d '{"email":"user@example.com"}'
   ```
3. If they forgot their password, direct them to `/forgot-password`.
4. If MFA is locked out, use a recovery code. If recovery codes are lost, an admin can reset MFA:
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
     "UPDATE identity.mfa_secrets SET status='revoked' WHERE user_id='<uuid>';"
   ```
   (User will need to set up MFA again on next login.)

### 10.2 "A user reports a bug"

1. Reproduce the issue yourself if possible.
2. Ask the user to submit feedback via the floating button (this captures context).
3. Look up the feedback:
   ```bash
   curl -sf "https://app.masteryengine.com/api/v1/beta/feedback" \
     -H "Authorization: Bearer $ADMIN_TOKEN" \
     | jq '.[] | select(.user_id=="<uuid>") | select(.created_at > "2026-07-01")'
   ```
4. Check Sentry for the same time window.
5. Check `correlation_id` (auto-captured) — use it to trace the request through logs.
6. File a bug; respond to the user with an ETA.

### 10.3 "A user wants to change their email"

There's no UI for this yet. The flow:

1. User requests email change → admin creates a verification token manually.
2. Or: user creates a new account with the new email, admin transfers their progress.
3. Track as P2 feature.

### 10.4 "A user wants their data deleted (GDPR)"

```bash
# 1. Anonymise the user's identity data
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  UPDATE identity.users
  SET email='deleted+' || id || '@anonymised.local',
      display_name='Deleted User',
      password_hash='DELETED',
      role='user',
      updated_at=now()
  WHERE id='<uuid>';
"

# 2. Delete their learning data (or anonymise)
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  DELETE FROM learning.attempts WHERE user_id='<uuid>';
  DELETE FROM learning.study_sessions WHERE user_id='<uuid>';
  DELETE FROM mastery.mastery WHERE user_id='<uuid>';
  DELETE FROM identity.beta_feedback WHERE user_id='<uuid>';
"

# 3. Revoke all tokens
docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c "
  UPDATE identity.refresh_tokens SET revoked_at=now(), revoke_reason='GDPR deletion' WHERE user_id='<uuid>';
  UPDATE identity.sessions SET revoked_at=now(), revoke_reason='GDPR deletion' WHERE user_id='<uuid>';
"
```

⚠️ Test this on a staging instance first. Track GDPR compliance as a P1 legal requirement.

### 10.5 "Beta users are reporting the platform is slow"

1. Check Grafana for p95 latency spikes.
2. Check Postgres for slow queries:
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
     "SELECT query, mean_exec_time, calls FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;"
   ```
3. Check Redis memory:
   ```bash
   docker compose -f docker-compose.prod.yml exec redis redis-cli -a "$REDIS_PASSWORD" info memory | grep used_memory_human
   ```
4. Check worker backlog:
   ```bash
   docker compose -f docker-compose.prod.yml exec postgres psql -U mastery -d mastery_engine -c \
     "SELECT count(*) FROM infrastructure.outbox_events WHERE consumed_at IS NULL;"
   ```
5. Follow `troubleshooting.md` for the specific bottleneck.
